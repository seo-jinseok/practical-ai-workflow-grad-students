### 프로젝트 개요
- **목표**: 박사 과정 입학 또는 연구비 신청
- **유형**: 연구 제안서 (Research Proposal)
- **기간**: 2주
- **전공**: 범용

### 통합 워크플로우 (압축 버전)

**Day 1-2: Context & Planning**
```markdown
# 연구 제안서 프로젝트

## 목표
[대학/기관] [프로그램/과제]에 제출할 연구 제안서

## 요구사항
- 분량: 10-15페이지
- 포함: 연구 배경, 질문, 방법, 예상 결과, 의의
- 마감: 2주 후

## 연구 아이디어
[간단한 아이디어 설명]

## 요청
1. 이 아이디어를 정교화해줘
2. 연구 제안서 작성을 위한 2주 계획
3. 각 섹션별 작업 분해
```

**Day 3-10: 실행**
- 각 섹션을 AI와 협업하여 작성
- 매일 1-2개 섹션 초안 완성
- 지속적으로 AI 검토 및 개선

**Day 11-14: 검증 및 완성**
- 전체 통합 및 일관성 확인
- AI로 전체 검토
- 동료/교수 피드백
- 최종 수정 및 제출

## 워크플로우 맞춤화 가이드

### 프로젝트 규모별 조정

**소규모 (1-2주)**:
- Step 1-2를 1일로 압축
- 작업 분해를 일 단위로
- 간소화된 검증 (AI + 자가)

**중규모 (1-3개월)**:
- 표준 워크플로우 적용
- 주간 루틴 확립
- 월간 마일스톤

**대규모 (6개월 이상)**:
- 상세한 Spec 문서
- 주간 + 월간 루틴
- 분기별 대규모 검토
- 위험 관리 계획 포함

### 전공 특성별 조정

**이론 중심 (인문학, 철학)**:
- Context 설정에 더 많은 시간
- 개념 정의 및 논리 구조에 집중
- AI로 논리적 일관성 검증

**실험 중심 (자연과학, 공학)**:
- 실험 프로토콜 상세화
- 데이터 분석 코드 작성에 AI 활용
- 재현성 검증 강화

**창작 중심 (예체능)**:
- 아이디어 브레인스토밍에 AI 활용
- 이론적 논거 개발 지원
- 기술적 문제 해결 도움

### 협업 프로젝트 조정

**공동 연구 시**:
- Spec 문서를 팀 전체가 공유
- 역할별 작업 분해
- 주간 미팅에서 AI 활용 내역 공유
- 통합 검증 프로세스

## 실전 팁 모음

### Context Engineering 팁
1. **템플릿 활용**: 자주 쓰는 프롬프트는 템플릿으로 저장
2. **컨텍스트 누적**: 대화 히스토리를 파일로 관리
3. **예시 라이브러리**: 좋은 결과물을 예시로 모아두기

### Planning 팁
1. **작은 단위**: 작업은 2시간-2주 크기가 적절
2. **버퍼 포함**: 예상 시간의 1.5배로 계획
3. **유연성**: 주간 계획은 고정, 일일 계획은 유연하게

### AI 활용 팁
1. **구체적 질문**: "도와줘" ❌ → "[구체적 작업] 해줘" ✅
2. **단계적 접근**: 한 번에 모든 것 ❌ → 단계별로 ✅
3. **피드백 제공**: AI 결과에 피드백하여 개선

### 시간 관리 팁
1. **AI 시간 제한**: AI와 대화는 하루 1-2시간으로 제한
2. **집중 시간**: 실제 작업(읽기, 쓰기, 분석)에 더 많은 시간
3. **루틴화**: 매주 같은 시간에 AI와 계획 점검

### 품질 관리 팁
1. **냉각 기간**: 작성 후 1-2일 뒤 재검토
2. **다층 검증**: AI → 자가 → 동료 → 교수
3. **체크리스트**: 분야별 품질 기준 체크리스트 활용

## 문제 해결 시나리오

### 문제 1: AI가 엉뚱한 답변

**상황**: AI가 연구 질문과 무관한 내용 생성

**해결**:
```markdown
# 재요청
이전 답변이 내 연구 질문과 맞지 않습니다.

## 내 연구 질문
[명확히 재진술]

## 필요한 것
[구체적으로 명시]

## 제약 조건
[추가 제약]

위 내용을 바탕으로 다시 작성해줘.
```

### 문제 2: 작업이 예상보다 오래 걸림

**상황**: 계획한 1주 작업이 2주째 진행 중

**해결**:
```markdown
# 현재 상황
- 작업: [작업명]
- 계획: 1주
- 실제: 2주째 진행 중 (60% 완료)
- 원인: [예상 원인]

## 요청
1. 이 작업을 완료하기 위한 전략
2. 전체 일정 조정 방안
3. 우선순위 재조정 필요 여부
```

### 문제 3: 연구 방향 변경 필요

**상황**: 중간에 연구 방향 수정 필요

**해결**:
```markdown
# 변경 사항
- 기존 방향: [설명]
- 새 방향: [설명]
- 변경 이유: [이유]

## 현재 진행 상황
- 완료한 작업: [목록]
- 진행 중인 작업: [목록]

## 요청
1. 완료한 작업 중 재사용 가능한 것
2. 새로운 작업 계획
3. 일정 영향 분석
4. 위험 요소 및 대응 방안
```

## 사용 가이드

### 시나리오 선택
- **강의 시**: 시나리오 1 (석사 논문) 메인 사용
- **질문 대응**: 다른 시나리오로 추가 설명
- **배포 자료**: 모든 시나리오 제공

### 맞춤화 방법
1. 자신의 전공/상황에 맞는 시나리오 선택
2. [대괄호] 부분을 자신의 정보로 교체
3. 불필요한 단계 생략, 필요한 단계 추가
4. 자신만의 템플릿으로 발전

### 지속적 개선
- 사용하면서 효과적인 프롬프트 기록
- 자신만의 프롬프트 라이브러리 구축
- 성공/실패 사례 메모
- 워크플로우 지속적 개선
